# 2026-06-25 - El primer greenfield real revela lo que el harness nunca construyo

> Bitakora.ControlAsistencia se convirtio en el banco de pruebas que ningun test unitario reemplaza: al correr el greenfield completo por primera vez, el planner descubrio que el harness promete siete modulos Terraform que nunca existieron, y que casi todos los ganchos del flujo CI/CD apuntaban al vacio.

## Lo que se logro

- **Release v0.7.0 publicado** (commits `29e1106`, `1441519`): primera version que empaqueta el soporte completo de greenfield Terraform — `bootstrap-backend.sh`, rutas corregidas en `infra-bootstrap`, seccion README de primeros pasos, y el mecanismo de copia+commit del `backend.tf` en el worktree del pipeline IaC.
- **Issues #83 y #86 cerrados con sus PRs** (#85 y #87): el #83 instalo el script que nunca existio (`bootstrap-backend.sh`), corrigio las rutas relativas invalidas de `agents/infra-bootstrap.md` y documento los primeros pasos greenfield en el README. El #86 cerro el seam que el #83 solo documento: el `backend.tf` ahora se copia y se commitea en la rama del worktree del pipeline IaC para que `terraform init` encuentre el backend remoto desde el primer plan.
- **Issue #90 creado y cerrado el mismo dia** (PR #91): el planner detecto que el scaffold de dominio emitia GitHub Actions desactualizadas (`checkout@v4` cuando hay `v7`, `setup-dotnet@v4` cuando hay `v5`, `azure/login@v2` cuando hay `v3`) y los bumpo en una sola sesion, de planificacion a merge en horas.
- **Backlog completo del primer greenfield real capturado y refinado**: 9 hallazgos de campo de `Bitakora.ControlAsistencia` (A-I) procesados por el planner en la sesion de las 11:33, resultando en 8 issues (los #92-#99), todos en `estado:listo` al cierre del dia. Incluye issues de infraestructura, CI/CD, documentacion y un agente generador nuevo.
- **ADR-0021 creado** (commit `f82c814`, PR #93 mergeado el dia siguiente): como parte del issue #93, el agente genero el ADR que formaliza la convencion de unicidad global para recursos de Azure (sufijo `random_string` de 6 caracteres) y la restriccion de region de PostgreSQL.
- **4 field notes del planner**: sesiones a las 04:56, 05:40, 10:45 y 11:33, que documentan el knowledge crunching y las decisiones de diseno tomadas antes de crear cada issue.

## Problemas encontrados

- **Los 7 modulos Terraform base no existian en el harness.** El `domain-scaffolder.md` (Paso 4) y el `infra-writer.md` asumen `infra/modules/` preexistente (`module.resource_group`, `.monitoring`, `.service_bus`, `.postgresql`, etc.), pero ninguna parte del harness los genera. ADR-0020 documenta el *contrato* del modulo `service-plan` (lineas 56-58) pero no provee la *plantilla*. Bitakora los habia escrito a mano; cualquier otro greenfield real hubiera tenido que reinventarlos desde cero. El hallazgo (B en la lista del planner) se convirtio en el issue #93, el mas complejo del dia.
- **El modulo `service-plan` de campo incumple el ADR-0020.** El modulo que Bitakora escribio a mano solo acepta `name/resource_group_name/location/sku_name/tags`; el scaffolder le pasa `os_type`, `worker_count` y `always_on` (contrato ADR-0020, `domain-scaffolder.md:1303-1306`). La plantilla de campo y el ADR estaban desfasados desde el dia en que se creo ADR-0020 (2026-06-18), sin que ningun test lo detectara.
- **El gate de issue OPEN bloqueaba el flujo preview -> merge -> apply.** `iac-pipeline.sh:206` aborta si el issue no esta `OPEN`, pero `iac-pipeline.sh:546` siempre emite `Closes #N` en el PR, incluso con `--skip-apply`. Al mergear el PR de preview el issue se cerraba automaticamente, dejando el apply posterior sin issue abierto que procesar — un deadlock operacional (issue #96).
- **El scope del plugin instalado a nivel project no cubre los worktrees.** `iac-pipeline.sh:332` corre `claude -p` dentro del worktree (`${REPO_ROOT}/../${BRANCH_NAME}`), pero un plugin instalado con `/plugin install` (scope project) no sigue al proceso hijo al directorio del worktree -> `agent not found`. El mismo patron afecta los pipelines TDD y scaffold (issue #95).
- **Mismatch de secrets entre el workflow del scaffolder y `setup-github-ci.sh`.** El deploy YAML generado por el scaffolder usa `azure/login` con `creds: ${{ secrets.AZURE_CREDENTIALS }}` (un JSON monolitico), pero `setup-github-ci.sh` crea 4 secrets separados (`AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, `AZURE_SUBSCRIPTION_ID`, `AZURE_CLIENT_SECRET`). El primer deploy de cualquier dominio scaffoldeado habria fallado con un secret inexistente (issue #97).
- **El workflow reusable `smoke-tests-dominio.yml` nunca se generaba.** El scaffolder lo *referencia* en el paso 6b y lo registra en `.github/smoke-tests-dominios.json`, pero ninguna parte del harness lo emite. El field note del #90 ya lo habia marcado como "fuera de alcance"; el greenfield real cerro ese gap con el issue #98.

## Lo que descartamos

- **Partir el #83 en 3 issues** (fix de rutas / script+backend / doc README): el planner lo propuso como primera lectura de la Revision de complejidad (>6 CAs, >1 componente). El usuario decidio mantenerlo monolitico con el argumento de que todos los CAs son facetas del mismo eje ("habilitar el arranque greenfield de Terraform") y asumio el alcance conscientemente.
- **Opcion (b) para el #86** (que el agente incluyera el `backend.tf` en el PR en vez de que el script lo copiara): dependia del agente `haiku` y era dificil de verificar. Se adopto la opcion (a) — copiar+commitear desde el script — por ser determinista y por subsumir la verificacion de campo sin depender del comportamiento del agente.
- **Solo copiar el `backend.tf` sin commitear (#86)**: resolveria el `terraform init` en la sesion local pero el archivo no apareceria en el diff del PR; el consumidor tendria que commitearlo aparte en un paso manual.
- **Directorio `templates/` copiable para los modulos base (#93)**: sin precedente en el harness (el `domain-scaffolder` no copia archivos, los emite inline), y congela contenido que debe ser dinamico (el `service-plan` debe cumplir ADR-0020, la region de Postgres depende del consumidor, los nombres necesitan sufijo de unicidad). Se adopto el patron del harness: agente generador que escribe contenido inline.
- **Crear un issue de robustez para que el pipeline detecte el scope incorrecto** (extension de #95): el planner lo anoto como decision abierta en el issue; el coordinador puede retomarlo. No se creo para no fragmentar el backlog con una mejora de segundo orden sin prioridad definida.
- **Anclar el tag de GitHub Actions por SHA** (en #90): el harness ya usa tags flotantes (`@vN`) y el pin por SHA es hardening del consumidor, fuera del alcance del template del scaffold.

## Aprendizajes

1. **El primer greenfield real es un audit de deuda oculta que ningun test ejercita.** Los 9 hallazgos de campo no surgieron de una revision de codigo: surgieron de intentar usar el harness de principio a fin. El gap de los 7 modulos Terraform existia desde el dia 1 del ADR-0020; solo se volvio visible cuando alguien corrio el flujo completo.
2. **"El ADR documenta el contrato" no garantiza que el codigo lo cumpla.** ADR-0020 especificaba los inputs de `modules/service-plan`, pero el modulo de campo (escrito a mano por Bitakora) solo tenia la mitad de ellos. La discrepancia entre el contrato declarado y la implementacion real no es visible sin correr el scaffolder contra un modulo real.
3. **Un issue monolitico es valido si las decisiones de diseno estan cerradas.** El #83 excedio el limite blando de 6 CAs de la Revision de complejidad del planner, pero las 3 decisiones abiertas se cerraron antes de emitir `estado:listo`. La complejidad de la implementacion es manejable cuando la ambiguedad de diseno es cero.
4. **El seam entre script y pipeline es una fuente de bugs sistematica.** El `bootstrap-backend.sh` escribia `backend.tf` en el working tree; el pipeline lo ignoraba porque ramifica desde `origin/main`. No era un bug del script ni del pipeline por separado — era un bug de la interfaz entre ambos, invisible hasta que se corre el flujo completo en greenfield.
5. **Fusionar issues de doc es una herramienta de gestion de complejidad valida.** El planner fusiono C + H en #99 (dos hallazgos de doc de bajo riesgo sobre region de Postgres y snippet de `harness.config.json`) e I en B (#93) como CA-4. No es una trampa contable: simplifica el backlog cuando los cambios son cohesivos y no generan conflictos entre si.
6. **Las versiones de GitHub Actions se desfasan en silencio.** No hay un mecanismo en el harness que avise cuando una version que emite el scaffolder queda atrasada. El hallazgo del #90 se detecto por inspeccion manual, no por un gate automatico.
