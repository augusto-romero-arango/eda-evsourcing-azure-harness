# 2026-07-08 - Cosmos.ControlPlane pone a prueba el harness: once costuras y un Key Vault que ya no se siembra a mano

> El primer scaffold paralelo de 5 dominios contra un consumidor real (Cosmos.ControlPlane / SincoERP) rompio el harness de once formas distintas en la misma tanda -- desde un `REPO_ROOT` que no sobrevivio la migracion a marketplace hasta un `ForbiddenByRbac` que bloqueaba al propio Owner de la suscripcion -- y el dia se paso resolviendolas todas, cerrando con la custodia de secretos trasladada por completo a CI y el scaffold de dominios ya paralelizable sin conflictos.

## Lo que se logro

- **`REPO_ROOT` corregido en `scaffold-pipeline.sh`** (issue #228, PR #230, 15:54-16:04 local): en instalaciones via marketplace `SCRIPT_DIR/..` apuntaba al cache del plugin (`~/.claude/plugins/cache/...`), que no es un repo git -- los 5 dominios de la tanda de campo murieron identicamente con `fatal: not a git repository`. Se alineo con `git rev-parse --show-toplevel`, el mismo patron que ya usaban los otros cinco pipelines publicados; `scaffold-pipeline.sh` era el unico rezagado desde la migracion de ADR-0019.
- **Reforma completa de la custodia de secretos: CI siembra TODO, ningun humano necesita rol de datos de Key Vault** (issue ancla #229 + hijos #231/#232/#233, PRs #235/#237/#239/#240, 16:20-16:56 local). Origen: el mismo consumidor, siguiendo el README paso 5 al pie de la letra, choco con `ForbiddenByRbac` al intentar `az keyvault secret set` **siendo Owner de la suscripcion** -- el control plane de Azure no otorga acceso al data plane de un Key Vault con RBAC habilitado, y ningun paso del bootstrap le asignaba `Key Vault Secrets Officer`. La correccion no fue un parche puntual sino invertir la doctrina completa (ADR-0025 decision #6/#10, ADR-0022):
  - **#229** reformo ambos ADR fijando el mecanismo M1: el propio SP de CI (que ya tiene `Role Based Access Control Administrator`) se auto-asigna `Key Vault Secrets Officer` sobre su vault durante el `apply`; ningun humano necesita ya ese rol.
  - **#231** lo materializo en `infra-base-scaffolder`: `infra-cd.yml` gana un step post-apply que siembra con `az keyvault secret set` la cadena de ASB interna, `marten-connection`, `app-insights-connection` y cada alias de `serviceBus.external[]` (alimentado por un GitHub secret propio, `SB_EXTERNAL_<ALIAS>_CONNECTION_STRING`), con reintento ante la latencia de propagacion RBAC.
  - **#232** actualizo `/onboard` para dejar de instruir la siembra manual y en su lugar diagnosticar (informativo, no bloqueante) si existen los GitHub secrets que la alimentan.
  - **#233** reescribio el paso 5 del README: la unica accion manual del admin pasa a ser crear esos GitHub secrets, nunca tocar el Key Vault.
- **Scaffold de dominios paralelizable sin conflictos entre archivos compartidos** (issue #234, PR #243, 17:12-17:18 local): la misma tanda de 5 dominios solo abrio 2 PRs limpios porque las 5 ramas partian del mismo `origin/main` y todas apilaban al final de `infra/environments/dev/main.tf`. Cuatro frentes en un solo issue grande (decision explicita del usuario de no desglosar): Terraform por dominio en archivos planos `dominio-<kebab>.tf` (verificado contra HashiCorp: Terraform no recorre subdirectorios, asi que un `domains/` habria exigido convertir cada dominio en modulo -- descartado), smoke tests registrados por dominio en `.github/smoke-tests/<kebab>.json` con la matrix armada por glob en vez del array compartido `.github/smoke-tests-dominios.json`, el `.slnx` aceptado como residual benigno (add/add trivial, sin cambio de codigo), y el log del pipeline con sufijo de PID para no colisionar entre corridas simultaneas. ADR-0013 registro el cambio del workflow global en su control de cambios.
- **Commit defensivo en `scaffold-pipeline.sh`** (issue #236, PR #244, 17:20-17:25 local): en la misma tanda, Billing genero sus archivos correctamente pero el Paso 8 del agente (LLM, no determinista) omitio el `git commit`, y `gh pr create` fallo con "No commits between main and ...". El pipeline ahora commitea defensivamente si el worktree quedo con cambios sin commitear -- restaurando antes `.claude/settings.json` para que el patch runtime no viaje en el commit -- y aborta con mensaje claro antes del push si de verdad no hay nada que subir.
- **Investigacion del bootstrap no determinista y su fix** (issue #238, field note 16:56, PR #246; issue #241, PR #247, 17:32-17:40 local): la investigacion confirmo que el `.gitignore` **raiz** del consumidor no lo emite ningun componente del harness -- aparecia por improvisacion del LLM en cada corrida, con riesgo real de que `local.settings.json` (con `Password=postgres`, escrito por `domain-scaffolder` Paso 9) quedara versionado en una corrida e ignorado en otra. El workflow reutilizable `smoke-tests-dominio.yml`, en cambio, ya tenia plantilla byte-fija -- el riesgo ahi era solo deriva de transcripcion, no ausencia de fuente. Fix: nuevo Paso 2c en `infra-base-scaffolder` que emite el `.gitignore` raiz con contenido byte-fijo en la corrida unica de greenfield (elimina el add/add porque ya no hay dos ramas creandolo), y `domain-scaffolder` deja de borrar el guard per-proyecto de `func init` como blindaje adicional.
- **Auditoria de deprecaciones de cara a azurerm v5.0** (issue #242, PR #248, 17:46-18:02 local): `enable_rbac_authorization` renombrado a `rbac_authorization_enabled` en el modulo `key-vault` (el argumento sera removido en v5.0 del provider). La auditoria explicita del resto de modulos (pedida por el propio issue) encontro un segundo deprecado no reportado por nadie -- `daily_data_cap_notifications_disabled` en `azurerm_application_insights` (modulo `monitoring`), misma semantica invertida -- corregido en el mismo PR por ser la misma clase de deuda tecnica. ADR-0022 se sincronizo (citaba el nombre viejo en prosa).
- **Compuerta de longitud del nombre de Function App corregida: 60, no 32** (issue #245, PR #249, 18:02-18:44 local): `domain-scaffolder` rechazaba de raiz dominios validos declarados en `harness.config.json` (`TenantManagement`, `TenantProvisioning`) porque `func-{prefix}-{kebab} > 32` chars, y como el pipeline corre no interactivo (`claude -p ... bypassPermissions`) el mensaje "elige un nombre mas corto" no llegaba a nadie -- simplemente no se creaba nada. El 32 estaba mal atribuido: es el limite de truncado del **host ID** de Azure Functions, que solo colisiona cuando dos Function Apps comparten storage account -- algo que este marco nunca hace (Storage propia por dominio, ADR-0020). El limite real del recurso `Microsoft.Web/sites` es 2-60. De paso, el truncado de la Storage Account (limite real, 24 chars) dejo de ser una instruccion LLM-driven ("avisa al usuario y trunca hasta que quepa") y paso a una regla mecanica fija (primeros 13 caracteres del kebab + sufijo aleatorio de 6) -- la misma clase de no-determinismo que #238, resuelta antes de que se manifestara en campo.

## Problemas encontrados

- **Una regresion de empaquetado quedo invisible hasta el primer consumidor externo real.** `scaffold-pipeline.sh` llevaba desde la migracion de ADR-0019 (plugin fuera del repo consumidor) con un `REPO_ROOT` que solo funcionaba si el desarrollador corria el harness desde dentro de su propio clon -- nunca se probo una instalacion via marketplace hasta que Cosmos.ControlPlane lo hizo. Los 5 dominios de la tanda murieron todos por la misma causa antes de que nadie pudiera scaffoldear nada.
- **La doctrina de custodia de secretos (ADR-0025, cerrada apenas el 01-jul) tenia un footgun que solo se manifestaba en el camino feliz documentado.** El README paso 5 instruia exactamente lo que producia `ForbiddenByRbac`: un admin, aun siendo Owner de la suscripcion, no puede sembrar secretos en un Key Vault con RBAC habilitado sin un rol de datos explicito, y nadie se lo asignaba. No era un caso de borde: era el paso 5 tal como estaba escrito.
- **El Paso 8 (commit) de `domain-scaffolder` es no determinista por construccion (es un LLM), y el pipeline no lo garantizaba.** De 5 dominios en la misma tanda, unos commitearon y otros no, con el mismo agente y el mismo prompt.
- **Una compuerta interna del harness bloqueaba dominios validos sin decirle a nadie por que.** El limite de 32 caracteres nunca fue un limite real de Azure -- era un limite de host ID mal atribuido, arrastrado sin verificacion contra las naming rules reales.
- **Los warnings de deprecacion de `azurerm` llevaban acumulandose en silencio en cada `plan`/`apply` de cualquier entorno generado**, sin bloquear nada mientras el pin siguiera en `~> 4.0`, pero listos para romper el primer bump a v5.0.

## Lo que descartamos

- **Un subdirectorio `infra/environments/dev/domains/` por dominio (frente 1 de #234):** descartado tras verificar contra la documentacion de HashiCorp que Terraform no recorre subdirectorios (solo entran como `module` con inputs propios) -- habria exigido convertir cada dominio en un modulo Terraform completo, un refactor mayor sin necesidad real. Se prefirio el archivo plano `dominio-<kebab>.tf` en el mismo directorio del entorno.
- **Reforzar solo la instruccion de transcripcion del `.gitignore` raiz (guard "solo si no existe" con contenido LLM) en vez de emitirlo desde `infra-base-scaffolder`:** descartado en la investigacion de #238 -- sigue expuesto al add/add bajo scaffold paralelo salvo que la transcripcion sea perfectamente identica entre corridas. Se prefirio la corrida unica greenfield, que elimina el conflicto por construccion en vez de por disciplina.
- **Desglosar el issue #234 en sub-issues por frente:** descartado por decision explicita del usuario -- los cuatro frentes (Terraform, smoke tests, `.slnx`, log del pipeline) comparten origen (el mismo reporte de campo) y se implementaron como fases de un unico worktree.
- **Confirmar en campo si `smoke-tests-dominio.yml` diverge de verdad antes de tocarlo:** descartado por ahora -- la investigacion de #238 concluyo que, al ser plantilla byte-fija, el riesgo es de deriva de transcripcion (bajo, benigno si es identica) y no de ausencia de fuente; se reforzo la instruccion en vez de hoistearlo a `infra-base-scaffolder` como se hizo con el `.gitignore`.
- **Dar al SP de CI un rol de escritura mas amplio que `Key Vault Secrets Officer` sobre el vault propio:** descartado -- M1 (auto-asignacion acotada al vault, rol de datos no privilegiado) fue el unico mecanismo evaluado y adoptado; mantiene la condicion anti-escalacion intacta (solo bloquea `Owner`/`User Access Administrator`/`Role Based Access Control Administrator`).

## Aprendizajes

1. **Un consumidor real corriendo el camino feliz documentado es la unica auditoria que de verdad valida un greenfield.** Seis dias antes (06-jul) se cerro una auditoria categorica completa del onboarding; ninguna de sus seis correcciones toco `scaffold-pipeline.sh`, el paso 5 del README ni la compuerta de 32 caracteres. Los tres eran bugs reales, en el camino feliz, que solo aparecieron cuando alguien scaffoldeo 5 dominios de verdad.
2. **Un limite "de seguridad" sin cita a la fuente original tiende a sobrevivir mucho mas alla de su justificacion real.** El 32 del Function App y el `enable_rbac_authorization` deprecado compartian el mismo patron: nadie habia vuelto a verificar la premisa contra la documentacion oficial (naming rules de `Microsoft.Web/sites`, registro del provider `azurerm`) desde que se escribio.
3. **Auditar la superficie vecina cuando se corrige una instancia especifica sigue pagando dividendos.** El issue #242 pedia solo renombrar un argumento; la auditoria explicita que el propio issue exigia como criterio de aceptacion encontro un segundo deprecado (`daily_data_cap_notifications_disabled`) que nadie habia reportado.
4. **La misma clase de bug puede aparecer dos veces en el mismo dia sin ser el mismo bug.** El `.gitignore` raiz no determinista (#238) y el truncado de Storage Account no determinista (parte de #245) son ambos "el LLM improvisa contenido que deberia ser mecanico" -- pero el primero se resolvio emitiendo una fuente byte-fija desde una corrida unica, y el segundo reemplazando una instruccion en prosa por una regla aritmetica fija. Reconocer el patron compartido no implica que el fix se copie igual.
5. **Invertir una doctrina completa (quien siembra los secretos) resulto mas simple, y mas robusto, que parchear el sintoma puntual.** El fix minimo para #229 hubiera sido darle al admin el rol de datos que le faltaba; en cambio se traslado la siembra entera a CI, eliminando cualquier necesidad futura de que un humano toque el data plane del Key Vault.
6. **Declarar un issue deliberadamente grande (sin desglosar) sigue siendo una decision valida cuando los frentes comparten origen y worktree.** #234 excedio la revision de complejidad simplificada en tamano, pero sus cuatro frentes eran trazables por adelantado y se implementaron sin friccion como fases de un mismo PR.
