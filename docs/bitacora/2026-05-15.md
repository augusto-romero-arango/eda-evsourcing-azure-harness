# 2026-05-15 - El dia que Mefisto se miro al espejo

> Una sesion de planner que empezo como diseno de un skill de release termino revelando que el harness asumia demasiado de su primer consumidor; en un solo dia se auditaron, corrigieron y publicaron las deudas tecnicas, y el repo abrio sus puertas al mundo.

## Lo que se logro

- **PR #2 mergeado** (madrugada del 15): separacion fisica completa entre skills publicados (`commands/`, `agents/`, `scripts/`) y skills internos (`.claude/commands/`, `.claude/agents/`, `.claude/scripts/`), con guards defensivos cruzados, gate de scope y routing cross-repo limitado a drafts. El arco narrativo del dia parte de este merge.
- **Skill `/mefisto-release`** (issue #3, PR #15): skill interno nuevo que automatiza el ciclo de versionado del plugin — detecta la fase automaticamente (`prepare` o `publish`), parsea el `CHANGELOG.md`, bumpea `plugin.json`, crea rama `release/vX.Y.Z`, abre PR y publica el GitHub Release con las notas del CHANGELOG como cuerpo. 542 lineas de implementacion entre el skill y el script bash.
- **Ocho issues de limpieza post-separacion** (issues #4–#11, PRs #18–#25): la sesion de planner de las 20:48 identifico sistematicamente ocho categorias de deuda tecnica y las convirtio en issues independientes `estado:listo`, todos implementados y mergeados el mismo dia:
  - #4/PR#18 — ADRs citados con numeracion de Bitakora en lugar del harness.
  - #5/PR#19 — dominios y contratos hardcoded de Bitakora en `eda-modeler`.
  - #6/PR#20 — guards defensivos faltantes en los cuatro pipelines principales.
  - #7/PR#21 — guards defensivos faltantes en cuatro scripts auxiliares.
  - #8/PR#22 — anti-patron de `git push` directo a main en `historiador` y `fix-review`.
  - #9/PR#23 — slug de Mefisto hardcoded en `_pipeline-common.sh`.
  - #10/PR#24 — referencias rotas en skills `draft`, `health-check` y `show-flow`.
  - #11/PR#25 — alcance mal documentado de `tooling-investigator` y label `dom:tooling` fantasma.
- **`/mefisto-sequential`** (issues #13 y #14, PRs #16 y #17): motor de batch interno (`mefisto-batch-pipeline.sh`) + skill tmux para encolar y procesar issues `tipo:tooling` de Mefisto en secuencia con merge automatico entre pasos.
- **Fix pane tmux** (PR #12): `pane_id` en lugar de indices implicitos, evitando rotura en macOS/iTerm2.
- **Repo publico**: el repo se hizo publico en GitHub con secret scanning + push protection + dependabot security updates activados, LICENSE MIT anadida (PR #26) y topics configurados. Primer dia de vida publica de Mefisto.

## Problemas encontrados

- **ADRs descalibrados en produccion**: los agentes publicados (`planner`, `test-writer`, `reviewer`, `implementer`) citaban numeros de ADR del consumidor original (Bitakora) en lugar del harness. Por ejemplo, el DoR estaba como `0014-definition-of-ready` cuando en Mefisto es `0011`; la convencion de tests como `0022` cuando es `0016`; la senal de refactor como `0023` cuando es `0017`. Un consumidor nuevo que leyera esas referencias buscaria ADRs que no existen. Corregido en PR #18 con busqueda y reemplazo dirigido.
- **`eda-modeler` con identidad de Bitakora**: el agente listaba literalmente "Dominios del sistema: Depuracion, CalculoHoras, Programacion, Empleados" y contratos especificos de Bitakora. Para cualquier otro consumidor, el agente hubiera inventariado dominios ficticios. Se elimino toda enumeracion hardcoded; ahora el agente obtiene el contexto del dominio leyendo los archivos del consumidor (PR #19).
- **Push directo a main en el propio historiador**: ironia mayuscula. El agente encargado de registrar el historial del proyecto violaba la regla de entrega del mismo. `historiador.md` y `fix-review.md` ejecutaban `git push` a main sin rama + PR. Corregido en PR #22 anadiendo un guard de rama con instruccion explicita de crear PR si la rama activa es main.
- **`dom:tooling` prometido pero no provisionado**: `tooling-investigator` sugeria a los usuarios crear issues con label `dom:tooling` en el consumidor, pero `setup-github-labels.sh` no lo provisionaba. El consumidor que siguiera el consejo del agente obtenia un error de label desconocido. Resolucion tomada: eliminar la sugerencia del label (opcion A), no provisionar el label nuevo, ya que Mefisto interno usa `tipo:tooling` sin el prefijo `dom:` (PR #25).

## Lo que descartamos

- **Issue contenedor "limpieza general post-separacion"**: era la opcion obvia para agrupar los ocho problemas encontrados, pero contradice la convencion del marco (no epics ni issues padre). La relacion entre los ocho queda en esta bitacora y en la field note de la sesion, no en un issue sintetico.
- **Single-shot de `/mefisto-release` con commit directo a main**: aunque es la convencion comun en releasing (muchos proyectos si commitean el bump directo a main), descartamos la excepcion. Mantener la regla universal — toda edicion por rama + PR — simplifica la auditoria y permite revisar el bump de version antes de publicar. Costo: dos invocaciones al skill en lugar de una.
- **Un solo issue monolitico para `/mefisto-sequential`**: la estimacion superaba los 30 minutos y mezclaba motor (invocable desde shell para debug) con UX (tmux + skill). Partir en dos issues pequenos (#13 y #14, con dependencia explicita) mantuvo cada PR digerible y aporto valor independiente.
- **Modo sin tmux para `/mefisto-sequential`**: se evaluo invocar el script en foreground para simplificar. Descartado porque rompia la consistencia de UX con `/mefisto-tooling`, que si usa tmux. El usuario espera el mismo patron.
- **Fusionar guards de pipelines principales (#6) con guards de auxiliares (#7)**: aunque son la misma idea, los pipelines principales son la cara visible del harness y los auxiliares son mas perifericos. Mantenerlos en issues separados permitio priorizarlos y ejecutarlos en paralelo.
- **Skill `/mefisto-release` publicado (para consumidores)**: descartado por no aplicar. El versionado del propio plugin solo tiene sentido dentro del repo de Mefisto.

## Aprendizajes

1. **Un harness hereda el sesgo del primer consumidor.** Cuando se extrae tooling de un proyecto especifico, los artefactos llevan baked-in supuestos del origen — numeros de ADR, nombres de dominio, labels de issues. La separacion fisica (publicado vs interno) no basta si el contenido textual sigue referenciando el proyecto original. Hace falta una pasada explicita de portabilidad.
2. **La auditoria sistematica genera deuda visible antes invisible.** Ocho issues en ~8 minutos de sesion de planner. No porque los problemas fueran nuevos, sino porque la revision metodica (componente por componente, criterio por criterio) los hizo ineludibles. Sin ese recorrido, la deuda hubiera seguido silenciosa.
3. **Los guards defensivos son documentacion ejecutable.** Un agente sin guard puede invocarse en contexto incorrecto y producir errores confusos. El guard aborta con un mensaje explicito. Esto resulto en cuatro pipelines y cuatro scripts corregidos el mismo dia que se descubrio el problema.
4. **El anti-patron mas irnico del dia**: el agente que documenta la historia del proyecto violaba la regla de entrega del proyecto. La deteccion solo fue posible porque la revision cubrio todos los agentes, no solo los nuevos.
5. **ADR-0018 (regla de tres) como freno al over-engineering prematuro.** Al disenar `/mefisto-sequential` se evaluo extraer logica comun con el `/sequential` publicado. El ADR da cobertura explicita para aceptar la duplicacion inicial: refactorizar al common solo cuando duela, no antes.
6. **El dia que un repo se hace publico no es solo infraestructura.** Activar secret scanning + push protection antes de publicar no es burocracia: es la diferencia entre descubrir un secreto filtrado internamente o que lo descubra un crawler externo.
