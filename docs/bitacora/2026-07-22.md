# 2026-07-22 - WorkOS + APIM en el borde: nueve issues en un solo batch, y el paso 1.5 que volvió a morder

> Un solo field note de planning destrabó la familia completa de identidad y autenticación en el borde -- MEF-ADR-0032, dos agentes generadores, tres skills `install-*` y una enmienda a MEF-ADR-0028 -- lanzada en un único batch secuencial de nueve issues sin abortar, aunque el propio `/mefisto-sequential` volvió a morder a quien lo operaba: esta vez no por un bug de shell (como el 2026-07-15) sino por un extractor de dependencias que no distingue "depende de" de "consumido por".

## Lo que se logró

- **Knowledge crunching de una sesión completa (field note 16:59, mefisto-planner)**: el pedido inicial -- "crear un skill para instalar WorkOS" -- se convirtió en un mapeo de que WorkOS **no es una cosa, son cuatro capas**: (1) aprovisionamiento de identidad (`IIdentityProvider` + adapter), (2) custodia de la API key (patrón `/seed-secret`), (3) auth/authz en el borde (APIM + `validate-jwt`), (4) la etapa de tenancy. El planner exploró a fondo el repo del consumidor real (`Cosmos.ControlPlane`, que ya corre el patrón en producción con un ADR local propio, `ADR-0027` "Propuesto") en vez de partir de cero, y de ahí salió el desglose completo: ancla doctrinal (#336) + enmienda de tenancy (#337) + dos agentes generadores (#335, #338) + tres skills `install-*` (#339, #340, #342) + extensión de `/onboard` (#341) + un fix independiente de secretos (#343).
- **MEF-ADR-0032 fija la doctrina de edge-auth que el harness no tenía (#336 → PR #345, 17:21 local)**: grep vacío de `workos`/`apim`/`authkit`/`identity` en todo el marco, pese a que ControlPlane ya pagó el costo de aprenderlo en producción (~5 PRs y varios `apply` rotos). El ADR embebe el catálogo de trampas B1-B10 verificado contra Microsoft Learn (orden estricto de `validate-jwt`, `<base/>` prohibido en scope global, `cors` solo evaluado en el preflight `OPTIONS`) y fija la propagación claim→header (`user_email`→`X-User-Id`, `tenant_id`→`X-Tenant-Id`) con anti-spoofing.
- **MEF-ADR-0028 se enmienda: la transición (a)→(b) de tenancy deja de ser manual para todo camino (#337 → PR #346, 17:31 local)**: la sección 3 original (2026-07-19) asumía que el mapping claims→header siempre era project-specific. Con WorkOS+APIM ese mapping vive una sola vez en la política global del gateway -- no por dominio --, así que el resolver es genérico y la migración *sí* es automatizable. Nueva sección 4 acota el levantamiento del límite "manual" **únicamente** al camino WorkOS+APIM.
- **Agente `apim-gateway-scaffolder`, fiel al catálogo de trampas (#335 → PR #347, 17:54 local)**: genera dos módulos Terraform reusables (`api-management`, `apim-function-api`) con cada trampa B1-B10 documentada como comentario HCL inline -- nunca dentro del `xml_content`, porque el schema de `validate-jwt` rechaza comentarios XML interpuestos entre `openid-config`/`issuers`/`required-claims`. Verificado contra el Terraform Registry (`azurerm_api_management*`, provider 4.81.0).
- **Agente `workos-identity-scaffolder` (#338 → PR #348, 18:10 local)**: genera el puerto `IIdentityProvider` + adapter `WorkOsIdentityProvider` (get-or-create idempotente por `external_id`) sin duplicar la composición de DI de MEF-ADR-0029 -- detecta si el dominio ya separó su wiring (forma SPLIT post-#319) o sigue inline, y cablea en el lugar correcto en cada caso. Gate de verificación empírica: sin doc oficial de un SDK de terceros reverificable por reflexión, usa `dotnet build` contra el paquete restaurado como la forma más fuerte disponible; si falla, degrada a "proponer" sin cablear a ciegas.
- **Fix de rotación de secretos externos (#343 → PR #349, 18:25 local)**: `az functionapp restart` no refleja una rotación. Verificado contra la doc oficial que son **dos cachés independientes**: la caché de resolución de Key Vault references del plano de control (solo se refresca cada 24h, ante un configuration change, o ante un POST a `config/configreferences/appsettings/refresh`) y la caché en memoria del propio worker (solo se descarta ciclando el proceso). El fix aplica ambos mecanismos -- POST de refresh + `stop`/`start` -- nunca solo uno.
- **`/install-workos` (#339 → PR #350, 18:39 local)**: guía el dashboard de WorkOS (incluyendo el hallazgo no obvio de crear explícitamente el rol `admin`, que WorkOS no trae por defecto) y cablea el adapter + la custodia de la API key vía `/seed-secret`, sin reimplementar su lógica.
- **`/install-apim` (#340 → PR #351, 18:56 local)**: invoca al agente APIM y ejecuta la transición (a)→(b) completa -- detecta el registro exacto que emite `domain-scaffolder`, lo reemplaza por `AgregarTenantResolverHibrido()`, y corre el gate de MEF-ADR-0029 (test de composición DI) antes de commitear cada dominio, degradando puntualmente sin abortar el resto del batch si un dominio no matchea el patrón esperado.
- **`/onboard` presenta la bifurcación explícita de dos caminos (#341 → PR #352, 19:08 local)**: crecer (auth desde el inicio, etapa `multi-tenant-header`) vs POC (sin auth, etapa `mono-tenant-transitorio` default) -- mapeo 1:1 a las dos etapas de MEF-ADR-0028, manteniendo `/onboard` como doctor de solo lectura.
- **`/install-auth`, el orquestador final (#342 → PR #353, 19:23 local)**: encadena `/install-workos` → `/install-apim` con un gate humano bloqueante en medio (verifica que `WORKOS_CLIENT_ID`/`WORKOS_API_KEY` ya existan antes de tocar APIM). Stateless: no reimplementa la idempotencia de los sub-skills, delega y propaga su reporte.
- **Batch secuencial completo + release v0.17.0**: los nueve issues se lanzaron con `/mefisto-sequential 336 337 335 338 343 339 340 341 342` -- doctrina primero, agentes en paralelo lógico, fix independiente intercalado, skills de capa, extensión de onboard, orquestador al final. Los nueve PRs (#345-#353) mergearon a `main` sin abortar el batch (salvo el override manual del paso 1.5, ver abajo). Release **v0.17.0** (minor): dos agentes nuevos, tres skills nuevos, un ADR nuevo, una enmienda y un fix de infraestructura, todo aditivo.

## Problemas encontrados

- **El propio `/mefisto-sequential` volvió a morder a quien lo estaba operando, en la misma corrida que lanzó el épico de auth**: el paso 1.5 (validación de dependencias) abortó reportando que **#335 dependía de #340 y #343**. La causa: el extractor de la sección `## Dependencias` captura *cualquier* referencia `#NNN` sin distinguir la dirección semántica -- la sección de #335 decía "Depende de #336" (forward real, válida), "Consumido por #340" (relación **inversa**: en realidad #340 depende de #335) y "La sección E se traslada a #343" (una nota, no una dependencia). El batch era topológicamente válido y sin ciclos, pero el chequeo abortó igual, obligando a un override manual del coordinador tras confirmar el orden a mano. De ahí salió el draft **#344**, documentando el caso real observado y dos ideas de fix (extractor dirigido por patrón `^- *(Depende de|Bloqueado por)`, o formalizar la sintaxis de `## Dependencias` en MEF-ADR-0007/MEF-ADR-0011). El propio issue ya adelanta que el mismo defecto (parseo direction-agnóstico) vive también en `scripts/pr-sync.sh` del lado publicado, con un failure mode distinto pero potencialmente peor: desbloqueo prematuro en vez de aborto conservador.
- **Costo real de no tener doctrina de edge-auth**: el ADR mismo documenta que ControlPlane pagó ~5 PRs y varios `apply` rotos en Terraform para llegar al patrón que hoy queda scaffoldeado de una sola vez -- el motivo original del pedido, no un problema nuevo del día, pero vale dejarlo explícito como la razón de fondo detrás de todo el batch.

## Lo que descartamos

- **Absorber el orquestador dentro de `/onboard`**: descartado -- `/onboard` se mantiene como doctor de solo lectura (diagnostica y, opt-in, escribe la elección de camino); la orquestación pesada sale a su propia pieza descubrible (`/install-auth`).
- **Un solo mega-skill "instalar WorkOS" con las cuatro capas**: descartado por tamaño y porque dejaba el gap de doctrina sin resolver -- se prefirió doctrina (ADR) + agentes separados + skills de capa + orquestador.
- **Nombres alternativos para el orquestador** (`/setup-auth`, `/onboard-auth`, `/auth`): descartados en favor de `/install-auth`, coherente con la familia `install-*` recién acordada.
- **Flip temprano de tenancy (antes de que el mapping fuera genérico) y flip tardío sin migración automática (mantener el status quo del ADR)**: ambos descartados frente a la opción validada -- automatizar la migración completa solo para el camino WorkOS+APIM, dejando el resto de caminos con el límite manual intacto.
- **Agentes fusionados con sus skills**: descartado -- agente de identidad (#338) y agente APIM (#335) se mantuvieron separados de los skills que los invocan (#339, #340) para permitir olas paralelas y revisiones limpias, mismo criterio que ya se usó en épicos anteriores.

## Aprendizajes

1. **Un IdP de terceros rara vez es "una cosa"**: descomponer WorkOS en sus cuatro capas (identidad, secreto, borde, tenancy) antes de escribir el primer issue evitó un mega-skill monolítico y permitió paralelizar agentes y skills por capa.
2. **Centralizar el mapping claim→header en el borde generaliza el resolver de tenancy.** La premisa de MEF-ADR-0028 (2026-07-19) de que ese mapping "siempre" era project-specific solo era cierta mientras no existiera una capa de borde que lo normalizara -- instalar WorkOS+APIM *es* literalmente el momento (a)→(b), y por eso la migración se pudo automatizar sin relajar el gate de MEF-ADR-0029.
3. **La doctrina "escribir local, aplicar en CI" desacopla la autoría del long-pole humano.** El código y el Terraform de los ocho issues no necesitaban ningún valor real de WorkOS para escribirse -- solo el `apply` final queda bloqueado por la creación de la cuenta, que es tarea del operador, no del pipeline.
4. **Un catálogo de trampas nacido de dolor real en producción (B1-B10, issue #335) es una espec más fuerte que una spec escrita desde cero.** Las ~5 PRs rotas de ControlPlane se convirtieron directamente en comentarios HCL inline del agente generado, sin reinterpretación.
5. **El paso 1.5 de `/mefisto-sequential` sigue siendo el eslabón más frágil del propio marco -- y ya van dos veces que muerde en vivo durante el lanzamiento del épico que se supone debía facilitar.** El 2026-07-15 fue un bug de semántica de shell (zsh sin word-splitting); hoy fue un extractor de dependencias direction-agnóstico. Síntoma idéntico (aborta o reporta falso éxito en medio de un batch real), causa raíz distinta cada vez -- lo que sugiere que el paso necesita su propio conjunto de casos de prueba, no solo parches puntuales al síntoma que se observó esa vez.
6. **Formalizar la sintaxis de `## Dependencias` (forward vs inversa vs nota) es deuda pendiente, no solo de `/mefisto-sequential`.** El propio #344 ya identificó que `scripts/pr-sync.sh` publicado comparte la misma raíz con un failure mode más peligroso (desbloqueo prematuro); quedó explícitamente para el refinamiento decidir si se corrige en paralelo.
