# 2026-06-30 - El implementer que nunca supo que habia dos namespaces

> La cadena de dos namespaces de ASB cerro su ultimo gap operativo: el implementer, que habia quedado fuera del alcance del scaffold inicial, ahora enruta topics y triggers al namespace correcto. El dia empezó con un analisis forense de los gaps, incluyo una correccion de severidad que demostró la diferencia entre "el doc dice mal" y "el deploy falla", y termino con nueve issues cerrados y cinco PRs mergeados antes de las 19:00.

## Lo que se logro

- **Nueve issues cerrados en un solo dia** (PRs #142, #143, #144, #149, #150, #151, #152, #153, #154): la jornada cerro la ola completa de la topologia de dos namespaces ASB iniciada el 2026-06-26. Los issues de la primera mitad del dia (#135, #128, #131, #130, #129) eran deuda de la cadena anterior; los de la segunda mitad (#148, #145, #146, #147) son los gaps del implementer detectados y refinados esta misma jornada.
- **ADR-0003 corregido** (issue #148, commit `09c1200`, PR #150): el ADR describía el wiring de dos namespaces como "dos llamadas a `HabilitarAzureServiceBusParaServerLess`", cuando el código real (verificado en el spike #129) es `HabilitarAzureServiceBusParaServerLess(interno)` para el broker default mas `AgregarAzureServiceBusNombradoServerless("integracion", integracion)` para el named broker. El lenguaje condicional del spike ("si #129 valida…") se convirtio en afirmativo.
- **Implementer actualizado para enrutar al namespace correcto** (issue #145, commit `bf8394a`, PR #151): `agents/implementer.md` ahora distingue si el evento es `IPublicEvent` (va al modulo `service_bus_integracion`) o privado (va al modulo `service_bus_interno`), y genera los bloques Terraform con los nombres de modulo correctos que introduce el #130. El caveat del helper bulk (`PublicarEventosServerless` captura privados y publicos por `IsAssignableTo(typeof(IEvent))`; usar por tipo, no por assembly) quedo documentado en el implementer.
- **Ejemplo de ServiceBusTrigger corregido en implementer.md** (issue #146, commit `6f1fde9`, PR #152): el ejemplo ilustrativo usaba `Connection = "ServiceBusConnectionString"`, desincronizado del app setting real `SERVICE_BUS_CONNECTION` que genera el domain-scaffolder. Higiene de documentacion, bajo riesgo operativo — pero una trampa real de copy-paste para cualquier implementador.
- **Wiring del lado consumidor disenado** (issue #147, commit `f737703`, PR #153): `agents/implementer.md` y `agents/domain-scaffolder.md` ahora instruyen al modelo para elegir el app setting correcto en el `[ServiceBusTrigger]` (`SERVICE_BUS_CONNECTION_INTERNO` para topics propios, `SERVICE_BUS_CONNECTION_INTEGRACION` para topics de otros dominios del BC) y documentan el alcance diferido del consumo inter-BC.
- **Infra-base-scaffolder y domain-scaffolder actualizados** (issue #130, commit `8559471`, PR #149): los scaffolders ya emiten el wiring completo de dos namespaces (modulos separados, app settings duales, RBAC por namespace), cerrando el gap del scaffold inicial que se habia detectado dias antes.
- **Field notes procesadas** (commit `a919562`, PR #154): las dos field notes de la jornada (investigacion transversal 11:20 y sesion del planner 12:49) entraron a main como registro de la sesion de diagnostico y planeacion.

## Problemas encontrados

- **G2 fue sobrediagnosticado como bug de runtime antes de verificar el codigo del consumidor.** La investigacion de las 11:20 identifico `Connection = "ServiceBusConnectionString"` en el ejemplo de `implementer.md` y lo clasifico como "[ALTA] connection string huerfano roto YA hoy". El usuario cuestionó la clasificacion ("funciona en mis ambientes de Azure"). Verificamos contra `Bitakora.ControlAsistencia/ControlHoras/FunctionEndpoint.cs`: usa `Connection = "SERVICE_BUS_CONNECTION"`, coherente con su `Program.cs` y el app setting que provisiona Terraform. El string erroneo vivía solo en el ejemplo del doc, no en el codigo generado. Resultado: G2 se reclasificó de "[ALTA] bug de runtime" a "[BAJA] higiene de documentacion"; el issue #146 perdio el label `bug`.
- **Ramas de documentacion obsoletas con contenido ya integrado en main.** Al cierre del dia se detectaron dos ramas docs (`docs/changelog-unreleased-0.3.0` y `docs/field-notes-refinamiento-30`) que apuntaban a commits anteriores al avance actual de main. Antes de eliminarlas fue necesario verificar a nivel de contenido (patch-id / diff) que su material ya estaba integrado, porque un merge apresurado habria revertido ~6000 lineas de trabajo de la ola de dos namespaces.

## Lo que descartamos

- **Draft cross-repo a Cosmos.BuildingBlocks por un listen helper de ASB**: el analisis inicial abrio la pregunta de si el paquete debia exponer un helper de subscribe (analogia con el `ListenToRabbitQueue` del RabbitMQ). Descartado — la "Rama A" (consumo via `[ServiceBusTrigger]` con seleccion de app setting por origen del topic) es suficiente, mas simple y evita una dependencia nueva en el paquete.
- **Particionar el issue #147 en epic + hijo**: el planner propuso separar el diseno del consumo dual en un issue padre (diseno) y un hijo (implementacion en el domain-scaffolder). Con la Rama A, el diseno y la implementacion caben en un issue unico sin ambiguedad; la particion solo habria fragmentado el backlog.
- **Re-inflar G2a a bug con label `bug`**: tras la verificacion contra el consumidor desplegado, no habia justificacion tecnica. Un ejemplo ilustrativo desincronizado merece `tipo:tooling`, no `bug`.
- **Crear un issue epic/contenedor para la cadena de dos namespaces**: la relacion entre los issues se mantiene via seccion `## Dependencias` en cada uno; un issue contenedor no anade informacion operativa y complica el cierre automatico via `Closes #N`.

## Aprendizajes

1. **Verificar hallazgos de "deployed broken" contra el codigo real del consumidor antes de clasificar severidad.** El analisis de la investigation reviso el doc del harness y el codigo del paquete, pero no contra un proyecto consumidor desplegado. La diferencia entre "el doc dice mal" y "el deploy falla" es la diferencia entre higiene y bug. Cruzar con el consumidor toma minutos y evita inflar issues innecesariamente.
2. **El helper bulk de ASB captura privados y publicos; el registro debe ser por tipo.** `PublicarEventosServerless(nombreConexion, topic, Assembly)` filtra por `IsAssignableTo(typeof(IEvent))`, no por `IPublicEvent`. Usarlo con el assembly completo rompe la separacion de namespaces en silencio: los privados se publican al broker de integracion sin error en runtime.
3. **El scaffold inicial y el flujo iterativo son dos codigos de entrega distintos.** La cadena #128/#130/#131 cubrio el provisioning y el publish del scaffold inicial, pero el flujo iterativo (un dominio que gana su primer evento publico despues del scaffold, via `implementer.md`) quedo sin actualizar. Cuando un issue es explicito en "NO toca X", hay que abrir un issue separado para X si X tambien necesita cambiar.
4. **Verificar a nivel de contenido (no de commit ID) antes de eliminar ramas viejas.** Una rama obsoleta cuyo HEAD es un commit anterior al avance de main puede parecer "integrada", pero si se mergea revierte el avance. La verificacion correcta es comparar el diff de la rama contra main, no asumir que la fecha del commit implica inclusion.
5. **El wiring real de dos namespaces ASB es asimetrico.** El broker default (`HabilitarAzureServiceBusParaServerLess`) maneja el namespace interno; el named broker (`AgregarAzureServiceBusNombradoServerless`) maneja el de integracion. Esta asimetria no es un detalle de implementacion: si se invierten, los eventos publicos se publican al namespace equivocado sin error visible en compile time.
