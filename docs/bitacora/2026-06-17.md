# 2026-06-17 - v0.3.0 sale a produccion y el campo refuta la teoria

> Publicamos la primera version que recoge la auditoria de portabilidad del 15 de mayo, pero al ejecutarla en un consumidor real descubrimos que un fix partido por la mitad se lee como completo, y que la solucion que daban por buena para la otra mitad no sobrevivia al primer experimento de campo.

## Lo que se logro

- **Release v0.3.0 publicado** (PRs #34, #35): tag anotado `v0.3.0` + GitHub Release. Es la primera version que empaqueta toda la limpieza post-separacion del 15 de mayo (skills internos, guards, routing de drafts, LICENSE MIT). El ciclo se ejecuto con `/mefisto-release` en sus dos fases (prepare -> merge -> publish).
- **Particion e implementacion del draft #30** (field note 15:37, PR #33): el draft originado desde el consumidor `Bitakora.ControlAsistencia` se dividio en dos issues por lado. #30 (scripts: `tmux-pipeline.sh` y `eda-lint.sh` resuelven el repo del consumidor con `git rev-parse --show-toplevel` en vez de `$SCRIPT_DIR/..`) se refino a `estado:listo`, se implemento y se mergeo. #31 (comandos) quedo diferido a un experimento de campo.
- **Reformulacion de #31 con solucion accionable**: tras refutar `${CLAUDE_PLUGIN_ROOT}` en campo, se reescribio el issue con una solucion de diseno determinada (hook `SessionStart` que escribe la ruta del plugin a `.claude/pipeline/.plugin-root`, con fallback al cache), se retiro `bloqueado` y paso a `estado:listo`.
- **Tres issues de tooling refinados e implementados** (field note 17:45, PRs #39, #40, #41), via `/mefisto-tooling 37` y `/mefisto-sequential 36 38`:
  - #37 / PR #39 - regla de estilo "condiciones en positivo" (`if(existe)` sobre `if(!existe)`) en `implementer.md` y `reviewer.md`.
  - #36 / PR #40 - recordatorio de `[Unreleased]` del CHANGELOG al final del pipeline interno (warning informativo, no gate duro).
  - #38 / PR #41 - Fase 5 de `/fix-review` publicado adaptada al modelo plugin (enrutar mejoras de agentes del harness a un draft).
- **Draft #36 capturado** sobre verificar el mantenimiento del CHANGELOG (luego refinado e implementado el mismo dia).

## Problemas encontrados

- **El CHANGELOG sin mantener rompio el release.** La fase prepare de `/mefisto-release` aborta si `[Unreleased]` esta vacio, y lo estaba: los 27 commits acumulados desde v0.2.0 nunca se habian registrado. Hubo que reconstruir las notas a posteriori en un PR aparte (#34) antes de poder versionar. Costo: un ciclo extra de rama + PR + merge antes del release real. De aqui salio el #36 como contramedida.
- **Un fix partido por la mitad se leyo como completo.** v0.3.0 incluia el arreglo de los scripts (#30, Defecto B) pero no el de los comandos (#31, Defecto A). Al correr `/parallel` en ControlAsistencia con v0.3.0 instalado, el agente se confundio: "el script ya no esta en ./scripts/ del repo... voy a localizar donde quedo el script ahora". El sintoma de campo era exactamente el Defecto A, abierto y diferido; la version publicada daba una falsa sensacion de cierre.
- **La solucion asumida para #31 no sobrevivio al experimento.** Daban por hecho que `${CLAUDE_PLUGIN_ROOT}/scripts/X.sh` resolveria las rutas de los comandos. El micro-experimento en el consumidor (tanto con el prefijo `!` como con la herramienta Bash del agente) devolvio `PLUGIN_ROOT=[]` en ambos contextos. La documentacion oficial lo confirma: la variable solo se inyecta en hooks, MCP, LSP y monitors, no en los bloques bash de un slash command. El agente tuvo que localizar el script por ruta absoluta en `~/.claude/plugins/cache/.../mefisto/0.3.0/scripts/`.

## Lo que descartamos

- **`${CLAUDE_PLUGIN_ROOT}` como solucion de #31**: refutado empiricamente. La direccion adoptada es un hook (donde la variable si esta disponible) que persiste la ruta a un archivo runtime que los comandos leen; el descubrimiento por glob en el cache queda como fallback documentado, no como solucion principal.
- **Bump `patch` para v0.3.0**: aunque se pidio `patch`, los cambios acumulados incluian funcionalidad nueva (skills internos, slug configurable, guards). Bajo SemVer estricto eso es `minor`. Se corrigio a `0.3.0`.
- **Crear un issue duplicado para el hallazgo de `/parallel`**: el sintoma ya estaba cubierto por #31. En vez de fragmentar el backlog, se dejo constancia como comentario de evidencia de campo en #31.
- **Saltar la precondicion del CHANGELOG vacio**: la regla del skill prohibe bypasses. Se resolvio poblando `[Unreleased]` via PR (#34), no forzando el release.
- **Gate duro para el check del CHANGELOG (#36)**: friccionaria docs y refactors (Keep a Changelog: no todo cambio es "notable"). Se implemento como warning informativo.

## Aprendizajes

1. **Publicar media solucion da falsa sensacion de cierre.** Partir #30 en dos issues fue correcto, pero el release intermedio que solo recogio una mitad hizo que el consumidor siguiera rompiendose con un sintoma que parecia resuelto. Cuando un fix se parte, conviene ser explicito sobre que mitad viaja en cada version.
2. **La documentacion no sustituye la verificacion empirica.** `${CLAUDE_PLUGIN_ROOT}` parecia la solucion obvia y la doc no la contradecia de frente, pero el experimento de campo la tumbo. El CA de "probarlo en un consumidor real" era irreductible; ningun analisis de codigo lo hubiera reemplazado.
3. **El CHANGELOG no mantenido se cobra el dia del release.** 27 commits sin registrar convirtieron un release de un comando en un ciclo extra de reconstruccion. La leccion se materializo en el #36: que el propio pipeline recuerde actualizar `[Unreleased]`.
4. **SemVer: las features acumuladas son minor aunque sean tooling interno.** El instinto de "es solo tooling, va patch" ignora que agregar capacidades backward-compatible es un minor por definicion.
5. **El reporte de un agente confundido es evidencia diagnostica, no ruido.** El comentario "voy a localizar donde quedo el script" no era una queja: era la confirmacion exacta del Defecto A y la pista (`cache/.../mefisto/0.3.0/scripts/`) que oriento la nueva solucion de #31.
